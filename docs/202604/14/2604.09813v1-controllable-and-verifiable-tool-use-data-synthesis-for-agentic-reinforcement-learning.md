## 摘要
现有的合成工具使用语料库主要针对离线监督微调（SFT）设计，而强化学习（RL）则需要支持奖励可检查的在线试错（rollouts）的可执行环境。我们提出了 COVERT，这是一个两阶段流水线：首先通过具有多级验证的自进化合成生成可靠的基础工具使用轨迹，然后应用保持真值（oracle-preserving）的增强手段，系统地增加环境复杂度。这些增强引入了干扰工具、间接或模糊的用户查询，以及噪声、多格式或错误的工具输出，同时严格保留真值工具调用和最终答案作为地面真值（ground truth）。这种设计使得标准案例可以通过参考匹配进行自动奖励计算，而对于错误检测等特殊行为，则通过轻量级裁判辅助验证，从而支持工具调用策略的 RL 优化。在 Qwen2.5-Instruct-14B 上，COVERT-RL 将 BFCL v3 的整体准确率从 56.5 提升至 59.9，ACEBench 从 53.0 提升至 59.3，且在通用能力基准测试上的退化极小；当在 SFT 基础上叠加时，准确率进一步达到 62.1 和 61.8，证实了叠加增益。这些结果表明，保持真值的合成环境提供了一个实用的 RL 精炼阶段，与 SFT 互补，可提高在模糊性和不可靠工具反馈下的工具使用鲁棒性。

## Abstract
Existing synthetic tool-use corpora are primarily designed for offline supervised fine-tuning, yet reinforcement learning (RL) requires executable environments that support reward-checkable online rollouts. We propose COVERT, a two-stage pipeline that first generates reliable base tool-use trajectories through self-evolving synthesis with multi-level validation, and then applies oracle-preserving augmentations that systematically increase environmental complexity. These augmentations introduce distractor tools, indirect or ambiguous user queries, and noisy, multi-format, or erroneous tool outputs, while strictly preserving oracle tool calls and final answers as ground truth. This design enables automatic reward computation via reference matching for standard cases and lightweight judge-assisted verification for special behaviors such as error detection, supporting RL optimization of tool-calling policies. On Qwen2.5-Instruct-14B, COVERT-RL improves overall accuracy on BFCL v3 from 56.5 to 59.9 and on ACEBench from 53.0 to 59.3, with minimal regressions on general-ability benchmarks; when stacked on SFT, it further reaches 62.1 and 61.8, confirming additive gains. These results suggest that oracle-preserving synthetic environments offer a practical RL refinement stage, complementary to SFT, for improving tool-use robustness under ambiguity and unreliable tool feedback.

#### 1. 核心问题与研究动机
*   **核心问题**：现有的工具调用（Tool-use）合成数据集主要为离线监督微调（SFT）设计，表现为“静态”的输入输出对。然而，强化学习（RL）需要一个“动态”的可执行环境，能够支持在线试错（Rollouts）并提供可验证的奖励信号。
*   **研究动机**：
    *   现实世界的工具调用日志稀缺且涉及隐私。
    *   现有模型在面对干扰工具、模糊查询、噪声输出或工具报错时鲁棒性不足。
    *   缺乏一种既能大规模合成数据，又能精确控制环境扰动并自动计算奖励的框架。

#### 2. 方法论：COVERT 框架
COVERT 采用两阶段流水线来构建 RL 就绪的合成环境：
*   **第一阶段：可靠基础轨迹生成**
    *   **自进化合成**：从少量种子数据开始，利用 LLM 生成新轨迹，通过验证后回填种子池，实现规模化扩张。
    *   **多级验证**：包含**格式检查**（JSON 结构）、**规则检查**（参数类型、枚举值约束）和**语义检查**（LLM 裁判评估工具选择与查询的匹配度）。
*   **第二阶段：保持真值（Oracle-Preserving）的增强**
    *   **核心思想**：修改环境组件（系统提示词、用户查询、工具输出），但严格保留原始的“真值工具调用（Oracle Call）”和“最终答案（Oracle Answer）”。
    *   **增强类型**：
        *   **精确可验证类**：引入干扰工具、重写模糊查询、改变输出格式或注入噪声。奖励通过参考匹配（Reference Matching）自动计算。
        *   **裁判辅助类**：模拟工具报错（如连接失败）或有问题的查询（如缺少参数）。奖励由轻量级 LLM 裁判根据预设的“真值标签”进行判定。
*   **奖励设计**：奖励函数由格式奖励（$r_{fmt}$）和正确性奖励（$r_{cor}$）加权组成。正确性奖励根据增强类型选择直接匹配或裁判评分。
*   **训练算法**：采用 **GRPO**（Group Relative Policy Optimization）算法，无需价值模型（Value Model），适合可验证奖励场景。

#### 3. 实验设计
*   **基准测试（Benchmarks）**：
    *   **工具调用**：BFCL v3（伯克利函数调用排行榜）和 ACEBench。
    *   **通用能力**：LiveCodeBench（代码）、AIME/Math500（数学）、MMLU-Pro（知识）、IFEval（指令遵循）等，用于监测模型是否发生退化。
*   **对比方法**：
    *   Qwen2.5-Instruct（原始模型）。
    *   TOUCAN-SFT（在 150 万条真实 MCP 数据上微调的模型）。
    *   COVERT-RL（在合成环境上进行 RL 训练）。
    *   TOUCAN-SFT + COVERT-RL（叠加方案）。
*   **评估协议**：使用统一的“极简提示词（Minimal Prompt）”，不提供额外的行为指导，以测试模型的原生鲁棒性。

#### 4. 资源与算力
*   **硬件**：使用 **8 × NVIDIA H200 GPU**。
*   **时长**：
    *   7B 模型：单次 RL 训练约 **28 小时**。
    *   14B 模型：单次 RL 训练时长约为 7B 的 **1.6 倍**（约 45 小时）。
*   **裁判模型**：使用 Qwen2.5-Instruct-7B 作为轻量级奖励裁判。

#### 5. 实验数量与充分性
*   **实验规模**：涵盖了 7B 和 14B 两个参数量级，并在 2 个主流工具调用榜单和 8 个通用能力榜单上进行了全面测试。
*   **充分性**：实验不仅验证了 RL 的独立增益，还验证了其与 SFT 的叠加效应（Complementarity）。
*   **客观性**：通过在通用能力榜单上的测试，客观展示了 RL 训练对模型基础能力的影响（结果显示退化极小）。
*   **公平性**：所有对比模型均在相同的“极简提示词”环境下评估，避免了提示词工程带来的偏差。

#### 6. 主要结论与发现
*   **性能提升**：COVERT-RL 显著提升了工具调用准确率。在 14B 模型上，BFCL v3 提升了 3.4%，ACEBench 提升了 6.3%。
*   **鲁棒性增强**：在“无关工具检测”和“特殊错误处理”等细分维度上进步明显，证明了增强策略的有效性。
*   **SFT 与 RL 互补**：SFT 提供结构化先验，RL 进一步精炼语义正确性和鲁棒性，两者叠加效果最优（14B 达到 62.1% BFCL 准确率）。
*   **能力保留**：与 SFT 相比，RL 训练在提升工具能力的同时，更好地保留了模型的数学、代码和通用推理能力。

#### 7. 优点
*   **可控性强**：通过 Stage II 的增强算子，可以针对性地训练模型处理特定的边缘案例（如工具报错）。
*   **可验证性高**：保持 Oracle 的设计解决了 RL 训练中奖励获取难的问题，降低了对昂贵 LLM 裁判的依赖。
*   **实用性**：证明了合成环境可以作为模型部署前的有效“精炼”手段。

#### 8. 不足与局限
*   **长程规划限制**：目前的环境主要针对短到中程交互，尚未模拟具有复杂状态依赖的长程智能体规划任务。
*   **裁判偏差风险**：虽然引入了 Oracle 标签辅助，但“裁判辅助类”增强仍依赖 LLM 判定，可能存在潜在的评分偏差。
*   **环境真实度**：模拟环境虽然多样，但与真实生产环境中的复杂 API 交互（如状态持久化、并发冲突）仍有差距。
